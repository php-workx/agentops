# Learning: Marketplace Version Drift Is Silent

## What We Learned

The `.claude-plugin/marketplace.json` version fell 3 releases behind (2.23.0 while releasing 2.26.0). No CI check, pre-push gate, or release pre-flight validates plugin/marketplace version parity. The release skill checks package.json, pyproject.toml, Cargo.toml, and go version constants — but not Claude/Codex plugin manifests.

## Why It Matters

Plugin consumers see stale version numbers even when the skill content is current. Undermines trust in the distribution channel.

## Source

v2.26.0 release — caught during manual validation, not by automation.

# Learning: docs/CHANGELOG Pre-Commit Hook Is Header-Only

## What We Learned

The pre-commit hook that syncs `docs/CHANGELOG.md` from root `CHANGELOG.md` only copies version headers (e.g., `## [2.26.0] - 2026-03-15`), not the entry content underneath. This results in empty version entries in the docs copy that require manual fix post-commit.

## Why It Matters

Every release that uses the pre-commit hook for CHANGELOG sync ships an incomplete docs copy. The fix is always a follow-up commit, which pollutes the release tag if not retagged.

## Source

v2.26.0 release — code review agent caught the empty entry; manual fix + retag required.

# Learning: Section Headings Drift When Tables Expand

## What We Learned

When adding rows to a markdown table (BF6-BF9 to the bug-finding pyramid), the section heading above the table (`### Bug-Finding Pyramid (BF1-BF5)`) was not updated. The heading range became stale while the table content was correct. This was replicated across 3 copies (source, codex, embedded) because the same edit pattern was applied to all three.

## Why It Matters

Stale headings create confusion for consumers who scan headings without reading table content. The pattern of "edit table content but forget the heading" is a recurring documentation anti-pattern.

## Source

v2.26.0 release — caught by code review agent post-release.
